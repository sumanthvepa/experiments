# Generating Type Annotations for libraries

The parameterized library used in this project does not provide type
annotations. While this does not have any impact on running the code,
it does cause problems when mypy is doing static analysis on test
code.

To solve this problem we need to generate type annotations in a stub file.

See test/test_sample.py for how to generate an initial annotation stub.
But that is not sufficient. You need to manually create the type signatures
for the code.

What follows is an AI generated explanation of how to generate type
annotation stub files (Generated by Google Gemini)

Python stub files, with the `.pyi` extension, are essential for static type checkers like Mypy, Pyright, and PyCharm. They provide type hints for your Python code without including any runtime logic. This allows type checkers to analyze your code for type errors without actually executing it.

Here's a breakdown of how to write Python stub `.pyi` files:

### What are `.pyi` files used for?

  * **Type Hinting for Libraries:** They are crucial for libraries, especially those without inline type hints or written in C/C++. They define the public interface of a module, including classes, variables, and functions, along with their types.
  * **Backward Compatibility:** If you need to support older Python versions that don't have built-in type hinting features, `.pyi` files allow you to add type information without modifying the original source.
  * **Reducing Code Churn:** For existing codebases, adding type hints via `.pyi` files can minimize changes to the main implementation files.
  * **Documentation and API Definition:** They serve as a clear and concise definition of a module's API, which can be helpful for developers.

### Basic Syntax and Conventions

Stub files are syntactically valid Python files, but they omit implementation details.

1.  **File Naming:**

      * For a module `your_module.py`, the stub file should be `your_module.pyi`.
      * For a package `your_package/`, the stub file for its `__init__.py` would be `your_package/__init__.pyi`.

2.  **Omitting Runtime Logic:**

      * **Function Bodies:** Replace function bodies with an ellipsis (`...`) instead of `pass`.
        ```python
        # In your_module.py:
        def greet(name: str) -> str:
            return f"Hello, {name}!"

        # In your_module.pyi:
        def greet(name: str) -> str: ...
        ```
      * **Variable Initializers:** For annotated variables, you don't need to assign a value.
        ```python
        # In your_module.py:
        MY_CONSTANT: int = 123

        # In your_module.pyi:
        MY_CONSTANT: int
        ```
      * **Default Arguments:** You can use `...` for default arguments.
        ```python
        # In your_module.py:
        def send_message(msg: str, user: str = "guest") -> None:
            print(f"Sending '{msg}' to {user}")

        # In your_module.pyi:
        def send_message(msg: str, user: str = ...) -> None: ...
        ```

3.  **Imports:**

      * Imports in `.pyi` files work similarly to regular Python files. You'll often need to import types from the `typing` module.
      * Stub files can import from other stub files, but they are generally read directly by type checkers, not imported at runtime by the Python interpreter.

4.  **Classes:**

      * Define class signatures, including attributes and methods, with their respective type hints.
      * Method bodies are replaced with `...`.

    <!-- end list -->

    ```python
    # In your_module.py:
    class MyClass:
        def __init__(self, value: int):
            self.value = value

        def get_value(self) -> int:
            return self.value * 2

    # In your_module.pyi:
    class MyClass:
        value: int # You can define attributes with types here

        def __init__(self, value: int) -> None: ...
        def get_value(self) -> int: ...
    ```

5.  **Type Aliases:**

    ```python
    # In your_module.pyi:
    from typing import Union, List

    UserId = str
    Coordinates = tuple[float, float]
    DataList = List[Union[int, str]]
    ```

6.  **Protocols (for structural subtyping):**

    ```python
    # In your_module.pyi:
    from typing import Protocol

    class Readable(Protocol):
        def read(self) -> str: ...
    ```

7.  **`@type_check_only`:**

      * If you have types defined in the stub file that are *only* for type checking and not meant to be imported or used at runtime, you can mark them with `@typing.type_check_only`.

    <!-- end list -->

    ```python
    # In your_module.pyi:
    from typing import Protocol, type_check_only

    @type_check_only
    class _InternalProtocol(Protocol):
        def _internal_method(self) -> None: ...
    ```

### Placement of `.pyi` files

  * **Alongside the `.py` file:** The most common approach is to place the `.pyi` file in the same directory as its corresponding `.py` file. Type checkers will automatically pick it up.
  * **Dedicated `stubs` directory:** For larger projects or when stubs are maintained separately, you can put `.pyi` files in a dedicated directory (e.g., `myproject/stubs`). You might need to configure your type checker's search path (e.g., `MYPYPATH` environment variable for Mypy) to include this directory.
  * **Stub-only Packages:** For distributing stubs for a third-party library, you can create a separate "stub-only" package. These typically follow the naming convention `foopkg-stubs` for a package named `foopkg`.

### Automatic Stub Generation

Tools can help you generate initial `.pyi` files, though they often require manual refinement.

  * **Mypy's `stubgen`:** Mypy includes a `stubgen` tool that can automatically generate draft stub files for Python modules and C extension modules.
    ```bash
    stubgen -m your_module -o out/
    # This will generate out/your_module.pyi
    ```
    Generated stubs often default to `Any` for types, so you'll need to manually add more precise annotations.
  * **Pyright's `--createstub`:** Pyright also has a command-line option to create stubs.
    ```bash
    pyright --createstub your_package
    ```
  * **IDE Features:** IDEs like PyCharm offer features to create stub files for your own implementations or for external libraries.

### Best Practices

  * **Be precise with types:** The more accurate your type hints are, the more effective your type checking will be.
  * **Keep it minimal:** Stub files should only contain the public interface and necessary type information. Avoid including any private implementation details.
  * **Match the runtime API:** Ensure your stub file accurately reflects the actual functions, classes, and variables exposed by the runtime module.
  * **Consider contributing to `typeshed`:** If you write stubs for a widely used library, consider contributing them to the `typeshed` repository, which is a collection of high-quality stub files for popular Python libraries.

By following these guidelines, you can effectively write and maintain Python stub files to improve the type safety and maintainability of your Python projects.
